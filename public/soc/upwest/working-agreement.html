Scope of Work

A private, permission-first AI-agent environment, delivered in two parts: a fixed-scope build of the foundation, then optional ongoing engineering as needs grow.

    01

    Summary

    I'll set up a private AI-agent environment for UpWest in two parts. Phase 1 is a focused, fixed-scope build of the technical foundation. Phase 2 is optional, hourly help for the business-specific work we decide to add later — more agents, tasks, skills, controlled actions, maintenance, and improvements.

    Phase 1 is about getting the environment, access controls, integrations, self-service tools, and recovery basics working properly. It is not meant to deliver a complete catalog of UpWest workflows in the first month.

    02

    What we're building

    I'll start with three agents. Each has its own profile, role, instructions, approved users, tools, credentials, data sources, tasks, skills, schedules, and context boundary.

    01

    Deal Flow Agent

    Pipeline, sourcing, and CRM context for the approved team.

    02

    Operational Agent

    Day-to-day operational retrieval across approved sources.

    03

    Financial Agent

    Cap table and financial context, held behind a hard boundary.

    Gil only

    These first three are a starting point, not a permanent limit. I'll create reusable configuration and deployment patterns so we can define more agents later without rebuilding the environment. Every additional agent still needs a clear purpose, owner, users, permissions, data scope, and testing. Unless we add one to Phase 1 in writing, additional agents belong in Phase 2.

    03

    The hosted setup

    I will:

    • Provision a dedicated Linux VPS for UpWest.
    • Install and configure Hermes Agent.
    • Set up secure remote administration, network access, and encrypted connections.
    • Keep credentials and tokens outside prompts, tasks, skills, and ordinary logs.
    • Configure the initial agent profiles and persistent state.
    • Set up basic monitoring for availability, connectors, and scheduled jobs.
    • Put backup and recovery procedures in place.

    This is a practical private deployment. Formal compliance certification, external penetration testing, and a guaranteed 24/7 operational SLA are not included.

    04

    Identity & real separation

    For the Deal Flow, Operational, and Financial Agent profiles, I'll define and test their roles and instructions, authorized users, available tools and integrations, credentials and data scope, memory and context boundaries, and where their tasks, skills, and schedules are stored.

    I'll create an UpWest-specific identity and access setup that maps approved identities from each channel to internal UpWest users. The access model is simple:

    • Approved team members can use the Deal Flow and Operational Agents.
    • Gil can use all three agents.
    • Only Gil can use the Financial Agent, Carta connection, or financial Dropbox connection.
    • Administrative access does not automatically grant access to protected financial content.

    Before any restricted company information is retrieved, I'll test that unknown, disabled, and unauthorized users are denied.

    05

    Access & context

    The ways people can use it

    I'll configure and test two channels:

    • Email — approved employees can email the right agent and receive a reply from the approved agent mailbox.
    • Hermes web / desktop — authenticated users see and use only the agents they are allowed to access.
    A note on Claude Desktop

    I looked into a Claude Desktop integration. As it stands, it connects only through an MCP connector — so Claude can ask the agent, but that's a step removed from talking to the agent directly, and the experience isn't as clean. If direct conversation matters, we can discuss other platforms that support talking to the agent more naturally.

    Keeping each person's context in the right place

    I'll set up a user-context mechanism that gives the selected agent its own role and instructions, approved shared UpWest information, the requesting employee's permitted relationship context, and restricted information only when both the person and the agent are authorized.

    The point is to avoid silently exposing one employee's private history to another, and to keep financial context inside the Financial Agent boundary.

    06

    How we'll work together

    At the start, I'll work with the team to:

    • Confirm the initial 3–4 users, their approved email addresses and accounts, and which agents each person can access.
    • Confirm again that the Financial Agent is restricted to Gil.
    • Agree on the approved Gmail, Calendar, Affinity, Carta, and Dropbox scope.
    • Review the vendor plans, APIs, OAuth options, and export methods that are actually available.
    • Agree on a practical hosting, backup, retention, and monitoring approach.

    I'll work closely with the UpWest team and come into the office when it's useful for account setup, decisions, testing, or feedback — and be candid about blocked dependencies, security concerns, unexpected costs, and technical limits as they come up.

    07

    What's included in the first month

    All Phase 1 integrations and initial tasks are read-only. The agents aren't inherently read-only; this is a deliberate starting boundary while we test identity, permissions, data scope, behavior, and audit information. Every connection below depends on access the vendor supports under UpWest's account.

    Google Workspace Read-only

    • Approved Gmail mailboxes, labels, or messages.
    • Approved Google Calendars and events.
    • Retrieval & unauthorized-access testing.

    Affinity Read-only

    • Approved lists, organizations, people, opportunities, fields, and activities.
    • Retrieval testing against approved records.
    • A check that Phase 1 cannot modify Affinity.

    Carta Gil only

    • A supported API, connector, or export under UpWest's account.
    • Restricted to Gil through the Financial Agent.
    • Retrieval of an approved example dataset.

    Dropbox Gil only

    • Read-only access to financial folders approved by Gil.
    • Available only to Gil through the Financial Agent.
    • Verify Phase 1 cannot create, change, move, or delete files.

    Unsupported API reverse engineering is not included.

    08

    Tasks, skills & scheduled jobs

    I'll configure and test the tools authorized users need to:

    • Create and edit a task.
    • Create and edit a reusable skill.
    • Test a task or skill before wider use.
    • Enable, disable, and remove it.
    • Review the relevant change history and restore a prior approved version.
    • Create, test, pause, modify, or remove a scheduled job.

    I'll use neutral technical examples in Phase 1 to prove these mechanisms. Production business workflows and scheduled reports belong in Phase 2 unless we specifically include them in writing.

    After Phase 1, UpWest should be able to maintain straightforward tasks, reusable skills, and scheduled jobs without me — by following the documented process: define the instruction, choose the permitted agent and sources, test with approved examples, enable it, review the result, and disable or restore it when needed.

    That self-service ability doesn't mean every workflow is simple or safe without engineering help. A new integration, complex permission rules, sensitive information, an action that changes another system, or difficult debugging may need Phase 2 support.

    09

    Monitoring, backups & recovery

    I'll configure:

    • Basic agent availability monitoring.
    • Connector authentication and retrieval monitoring.
    • Scheduled-job status and failure reporting.
    • Basic visibility into model usage and operating costs.
    • VPS snapshots or off-server backups at the practical level we agree on.
    • Protected backups of agent configuration, approved memory, tasks, skills, schedules, and relevant state.
    • Version and rollback procedures for tasks, skills, and important configuration.
    • One agreed recovery test before handover.

    A backup restores lost state. A rollback returns a task, skill, or configuration to a previous working version. I'll address both.

    10

    A custom admin console

    Rather than managing everything through raw configuration files and the command line, I'll build a custom admin GUI for UpWest — a single place to see the environment's state and act on it without needing me in the loop.

    From the console, authorized administrators can:

    • Browse and edit tasks, skills, and scheduled jobs.
    • View change history and roll back to a previous approved version.
    • Manage users, their approved agents, and access levels.
    • Review agent, connector, and scheduled-job status at a glance.
    • Enable, disable, or remove a task, skill, or job.

    The console respects the same permission boundaries as the agents themselves — administrative access to the interface does not grant access to protected financial content.

    11

    Documentation & handover

    I'll provide straightforward documentation covering the environment and administrative access; initial users, agents, and permissions; access through each communication channel; integration scope and known limitations; managing tasks, skills, and schedules; monitoring, backup, recovery, and rollback procedures; and known open issues with recommended Phase 2 priorities.

    I'll also run an initial handover session for the first users.

    12

    What I'll hand over

    By the end of Phase 1, the deliverables are:

    1. A dedicated hosted Hermes Agent environment.
    2. Deal Flow, Operational, and Gil-only Financial Agent profiles.
    3. Identity mapping and initial access configuration for approximately 3–4 users.
    4. Configured email and Hermes access.
    5. Read-only Google Workspace and Affinity connections within the approved scope.
    6. Read-only Carta and Dropbox connections for Gil within the approved scope.
    7. Task, skill, version, rollback, and scheduled-job mechanisms.
    8. Monitoring, backup, recovery, and cost-visibility setup at the agreed level.
    9. Access, denial, connector, failure, and recovery tests.
    10. Documentation and an initial user handover.
    11. Reusable configuration patterns for future agents.
    13

    How we'll know Phase 1 is done

    Phase 1 is complete when these agreed checks pass — or when a third-party limitation is documented and accepted:

    • Approved users can reach their permitted agents through the agreed channels.
    • Unauthorized users cannot reach restricted agents or data.
    • Only Gil can access the Financial Agent, Carta, and financial Dropbox information.
    • Approved test data can be retrieved through supported Google Workspace, Affinity, Carta, and Dropbox connections.
    • Phase 1 integrations cannot modify connected source systems.
    • An authorized user can create, test, disable, and restore a test task and skill.
    • A test scheduled job can be run, paused, and reviewed.
    • Monitoring and backup status are visible.
    • An agreed recovery or rollback test is complete.
    • The documentation and known limitations are delivered.
    14

    Timing & dependencies

    I plan to complete Phase 1 within no more than one calendar month from our agreed kickoff date. The working sequence is:

    1. Confirm users, permissions, accounts, data scope, and vendor access.
    2. Establish the hosted environment and initial agents.
    3. Configure the communication channels and integrations.
    4. Configure tasks, skills, schedules, monitoring, backups, and rollback.
    5. Run access, retrieval, failure, and recovery tests.
    6. Fix identified issues, finish the documentation, and complete handover.

    That schedule depends on the required accounts, permissions, vendor capabilities, responsible people, and test information being available in time. If access is delayed or a provider has no supported connection method, I'll complete everything available and document the blocked item. An unavailable connector cannot be accepted as operational.

    What I'll need from UpWest

    • Relevant team members for questions, tests, and feedback.
    • Company email accounts for the agents.
    • The approved employee list and access assignments.
    • Administrator help for Google Workspace and other connected systems.

    • Gil's participation in setting up the Financial Agent, Carta, and Dropbox.

    • Decisions on hosting region, retention, mailbox scope, calendar scope, CRM scope, and Dropbox folder scope.
    • Timely review of acceptance results and known limitations.
    16

    What isn't included in Phase 1

    The fixed first phase does not include:

    • A complete production task or skill catalog.
    • Finished weekly or scheduled business reports.
    • Automatic source-system changes or external outreach.
    • Broad write access to connected systems.
    • Agents beyond the initial three.
    • Unsupported API reverse engineering.
    • A custom UpWest web or mobile application.
    • Detailed CFO, reconciliation, or investment-decision workflows.
    • LinkedIn or external data-enrichment integrations.
    • Formal compliance certification, legal privacy review, or external penetration testing.
    • 24/7 support or a guaranteed service-response SLA.
    • Third-party licenses, hosting, model, API, or connector charges.

    If we want to add something, I'll estimate it and we'll approve it before I begin.

    17

    What comes later — Phase 2

    Phase 2 is optional ongoing engineering, billed only for the time we actually use. UpWest can handle straightforward tasks, skills, and scheduled jobs internally after Phase 1, and bring me in for more complex work, new capabilities, troubleshooting, or ongoing support. It is not a mandatory retainer or an unlimited support commitment.

    Phase 2 may include:

    • Defining and deploying additional agents.
    • Building business-specific tasks and skills.
    • Scheduled reports and recurring workflows.
    • Controlled write or action capabilities.
    • Testing, debugging, and quality improvements.
    • Integration maintenance and authentication repairs.
    • New supported integrations.
    • Training and documentation updates.
    • Optimizing model usage and operating costs.

    For substantial requests, I'll share an estimate or working range before I start. My work records will identify the request, time used, changes made, tests completed, and known remaining issues.

    18

    Controlled actions, when we're ready

    Later, we may choose to let an agent take specific actions — for example: send an approved email, create or update an Affinity record, create a calendar event, move an approved document, or trigger another clearly defined system action.
